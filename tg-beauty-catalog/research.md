# Research: Telegram Mini App — Каталог услуг бьюти-мастера

> Исследование проведено: 13.04.2026  
> Метод: параллельный запуск 3 агентов по направлениям: примеры приложений / UI-паттерны / конкурентные преимущества

---

## 1. Обзор рынка

Полноценных Telegram Mini Apps в нише бьюти (именно WebApp внутри Telegram, а не боты с кнопками) — крайне мало. Рынок делится на три сегмента:

1. **Self-hosted open-source боты** — требуют разработчика, не доступны мастерам без техбэкграунда
2. **SaaS-платформы** (Beauty-Bot, Beauty-App.ru) — ориентированы на салоны, дорого для соло-мастера
3. **No-code конструкторы** (OkiBot, BotHelp) — быстро, но без реального Mini App UI и без синхронизации расписания

**Вывод:** Рынок ранний. Никто не совместил: портфолио + запись + депозит + CRM — в одном нативном Mini App.

## 2. Примеры приложений

### Пример 1 — nazgool97/telegram-salon-booking-bot (GitHub)

**Что делает:** Full Mini App на React внутри Telegram. Выбор услуги → мастер → слот → оплата → подтверждение. Три роли: клиент, мастер, администратор. Аналитика: LTV, процент no-show, выручка.

**Сильные стороны:**
- Единственный open-source проект с полноценным React UI внутри Telegram (не кнопки)
- Защита от двойного бронирования через `pg_advisory_xact_lock`
- Автоматические напоминания, интеграция Telegram Payments
- Аналитика для владельца: процент no-show, LTV клиентов, выручка факт vs план

**Слабые стороны:**
- Требует self-hosting (Docker + PostgreSQL) — слишком сложно для соло-мастера
- Нет публичного демо, нет портфолио/галереи работ
- Оплата только через Telegram Payments (без Yookassa/Stripe без кастомизации)
- Нет живого сообщества, код почти без документации

**Что мы можем сделать лучше:**
- Добавить галерею портфолио как центральный элемент
- Максимально упростить онбординг мастера (10 минут до live)
- Поддержать Yookassa и облачный хостинг без настройки сервера

### Пример 2 — Beauty-Bot (bot-beauty.ru)

**Что делает:** Коммерческий SaaS для российских салонов. Клиент переходит по ссылке → выбирает услугу → мастера → слот (синхронизация с календарём) → получает подтверждение и напоминания за 24ч и 2ч. Мультилокация, история клиента.

**Сильные стороны:**
- Синхронизация расписания в реальном времени
- Двойные напоминания (24ч + 2ч) — реально снижают no-show
- Доступная цена для старта: 990₽/мес для до 5 мастеров
- Работает из Instagram, 2GIS, QR-кода

**Слабые стороны:**
- Нет Mini App WebApp-интерфейса — классический бот с кнопками (no full-screen UI)
- Нет портфолио/галереи работ мастера
- Панель администратора — отдельный веб-сайт, нельзя управлять из Telegram
- Нет публичного демо

**Что мы можем сделать лучше:**
- Сделать полноценный WebApp с визуальным каталогом, а не кнопки
- Встроить портфолио прямо в карточку мастера/услуги
- Управление расписанием из Telegram (без выхода в браузер)

### Пример 3 — Beauty-App.ru (Telegram Mini App + YClients)

**Что делает:** Коммерческий SaaS, интегрирован с YClients CRM. Полноценный Mini App WebApp UI: онлайн-запись, программа лояльности, автоматические рассылки, управление загрузкой мастеров.

**Сильные стороны:**
- Настоящий WebApp-интерфейс (не кнопки)
- Интеграция с YClients — стандарт де-факто для салонов России/СНГ
- Программа лояльности внутри Telegram
- По данным компании: +30% записей, 76% подтверждений без участия администратора

**Слабые стороны:**
- Жёсткая зависимость от YClients — кто не платит за YClients, не может пользоваться
- Цена не публикуется — нужна консультация (барьер входа)
- Метрики (+30% записей) — маркетинговые заявления без кейсов и имён клиентов
- Нет публичного демо-бота
- Суммарная стоимость: YClients + Beauty-App.ru = дорого для соло-мастера

**Что мы можем сделать лучше:**
- Прозрачное и доступное ценообразование (flat fee)
- Не требовать сторонней CRM для базового функционала
- Опубликовать рабочее демо с реальными скриншотами

### Пример 4 — GrozNailTgBot (GitHub — бот для мастера маникюра)

**Что делает:** Open-source бот для одного мастера. Мастер добавляет слоты через скрытые команды. Клиент видит доступное время и бронирует. Мастер подтверждает или отклоняет нажатием кнопки. Клиент может отменить/перенести, просмотреть прайс.

**Сильные стороны:**
- Решает главную боль соло-мастера: убирает бесконечные «вы свободны в пятницу?» в Директе
- Accept/Reject от мастера — контроль без ручного труда
- Бесплатный хостинг на Vercel
- MIT-лицензия, простой код для форка

**Слабые стороны:**
- Нет Mini App WebApp — только кнопки (старый UX)
- Нет галереи/портфолио
- Только 1 мастер — не масштабируется
- Нет оплаты и напоминаний
- Устаревшая версия библиотеки (python-telegram-bot v13)

**Что мы можем сделать лучше:**
- Сделать визуальный Mini App UI поверх той же логики
- Добавить портфолио работ и оплату депозита
- Поддержать нескольких мастеров

### Пример 5 — OkiBot (no-code конструктор, шаблон для бьюти-салона)

**Что делает:** No-code шаблон: приветствие, меню (о нас, услуги, акции, контакты, запись), каталог услуг с фото и ценами, форма бронирования, оплата через Yandex.Kassa, рассылки подписчикам.

**Сильные стороны:**
- Запуск за 10 минут без кода
- Оплата внутри Telegram (Yandex.Kassa)
- Рассылки по подписчикам — реально полезно для акций
- Фото услуг в каталоге — шаг вперёд по сравнению с текстовыми ботами

**Слабые стороны:**
- Оплата требует оформления ИП/самозанятость + Yandex.Kassa — недоступно для «серых» мастеров
- Нет синхронизации календаря — форма заявки, а не реальная запись
- Нет напоминаний — после заявки бот замолкает
- Нет портфолио мастера (только фото конкретной услуги)
- Нет Mini App WebApp UI — кнопки + формы, без full-screen визуала
- Ограниченная кастомизация шаблона

**Что мы можем сделать лучше:**
- Добавить реальную синхронизацию слотов
- Полноценный WebApp UI вместо кнопок
- Портфолио мастера как отдельный раздел

## 4. Главные боли рынка

### Со стороны мастеров
- **Комиссии:** Booksy берёт 30%, Fresha — 20% за «своих» клиентов + 2.29% за транзакцию
- **Потеря данных:** при смене платформы клиентская база остаётся у агрегатора
- **No-show:** средние потери — $15,000–$67,000 в год на салон; ни одно CIS-решение не закрывает депозитом нативно
- **Ручная работа:** управление записями отнимает 12–20 часов в неделю
- **DIKIDI** (лидер в России): регулярные падения, плохая поддержка, глюки

### Со стороны клиентов
- Нельзя посмотреть портфолио мастера прямо в процессе записи — нужно прыгать в Instagram
- Реклама конкурентов в процессе записи (Booksy показывает других мастеров)
- Нет напоминаний в Telegram — теряются в email
- Нужно скачивать приложение чтобы записаться один раз

## 5. Что мы можем сделать лучше

### 1. Портфолио как главный экран (никто так не делает)
Все существующие решения ставят форму записи на первое место, а портфолио — опциональный раздел или ссылка на Instagram. Мы ставим работы мастера на главный экран: Instagram-style галерея с «Записаться на этот дизайн» прямо из фото.

### 2. Zero-комиссия, flat fee
Booksy/Fresha/YClients зарабатывают % с каждой записи. Фиксированная подписка (без комиссий за запись) — принципиально другая модель, которая выгоднее при любой загрузке мастера.

### 3. Депозит внутри Telegram (нативно)
Сейчас депозиты собирают через PayPal, банковский перевод или «пришлите реквизиты». Мы делаем `openInvoice()` с Apple Pay / Google Pay прямо в момент подтверждения слота. Депозит = подтверждение.

### 4. Запуск за 10 минут
5 фото + 3–5 услуг с ценами + рабочие часы → ссылка на Mini App готова. Никакой настройки сервера, никакого онбординга.

### 5. Клиентские карточки в Telegram
После визита мастер записывает заметки (аллергии, предпочтения, оттенок покрытия) прямо через бот-команду. Рассылка напоминаний и «пора на повтор» — в Telegram, где открываемость в 5–10x выше email.

## 7. Обязательные UI-элементы и паттерны

### Навигация
- **Bottom Tab Bar (4 таба):** Главная / Услуги / Мои записи / Профиль
- Минимальная зона тапа: 44×44pt (iOS) / 48×48dp (Android)
- Левый/правый отступ от края: не менее 16px (конфликт с iOS edge-swipe)

### Telegram-нативные элементы (обязательно использовать)
- `MainButton` / `BottomButton` — CTA на каждом шаге записи («Записаться», «Подтвердить», «Оплатить»)
- `BackButton` — нативная навигация, не делать свою кнопку «назад»
- `HapticFeedback` — лёгкий отклик на тапы, `notificationOccurred('success')` при подтверждении брони
- `ClosingConfirmation` — включить в шагах записи, чтобы пользователь не потерял прогресс
- `themeParams` — все цвета адаптируются к тёмной/светлой теме Telegram автоматически
- `CloudStorage` — сохранять черновик записи, просмотренные услуги
- `requestContact()` — получить телефон одним тапом (без регистрационных форм)

### Анти-паттерны (избегать)
- Потеря состояния при закрытии — пользователь прерывает запись, возвращается с нуля
- Горизонтальный свайп близко к краям — закрывает Mini App на iOS
- Spinner вместо skeleton-экрана — воспринимается как «зависло»
- Мелкие кнопки — пальцы, не курсор

## 9. Итоговые выводы

1. **Галерея работ отсутствует во всех найденных решениях** — при том, что бьюти — визуальная ниша. Это главный пропуск рынка.

2. **Соло-мастера не охвачены нормально:** SaaS-платформы дороги + ориентированы на салоны; open-source требует разработчика; конструкторы дают лид-форму, а не реальную запись.

3. **Telegram в России/СНГ — идеальная площадка:** 78.3% проникновения интернет-пользователей. Клиенты уже там. CityBite показал +420% заказов переходом в Telegram.

4. **Ни одно решение не закрывает комбинацию:**  
   WebApp UI + портфолио + реальный слот + депозит + CRM-карточки клиента — всё в одном нативном Telegram Mini App.

5. **Benchmark UX:** DurgerKingBot (официальный демо Telegram) + CityBite = как должен выглядеть каталог и флоу покупки. Применить к бьюти-нише.

## 10. Экспертная оценка

> Оценка проведена: 13.04.2026  
> Три угла зрения: UX-дизайнер / Маркетолог / Telegram-разработчик

---

### 10.1 UX-дизайнер

#### Типичные ошибки новичков в этой нише

**1. Каталог без фото — убийца конверсии**  
Новички делают список услуг текстом: «Маникюр — 1500₽». Бьюти — визуальная ниша. Клиент выбирает глазами. Без фото работ решение откладывается или не принимается вовсе. Каждая карточка услуги обязана иметь фото результата, а не иконку.

**2. Один длинный экран вместо шагов**  
Попытка уместить выбор услуги + мастера + даты + времени + контактов на одну страницу — главная ошибка флоу. Пользователь теряется, не понимает прогресса. Правило: один шаг = один экран. Максимум 4 шага до «Подтвердить».

**3. Потеря состояния при закрытии**  
Пользователь выбрал услугу, мастера, дату — закрыл Telegram чтобы уточнить что-то — вернулся на стартовый экран. Всё потеряно. Это раздражает сильнее всего. Черновик записи должен сохраняться в `CloudStorage` и восстанавливаться при следующем открытии.

**4. Мелкие кнопки и тесные элементы**  
Mini App открывается на телефоне, пальцами. Минимальная зона тапа — 44pt. Кнопки «Записаться» с высотой 28px — не кнопки, а мучение. Особенно раздражают маленькие переключатели времени в сетке слотов.

**5. Отсутствие пустых состояний**  
Если у мастера нет свободных слотов на этой неделе — показывают пустой экран или spinner-зависон. Нужно: иллюстрация + «Ближайшее свободное время — вторник 15 апреля» + кнопка «Перейти».

**6. Нет подтверждения действия**  
После нажатия «Записаться» — тишина. Пользователь не знает: записался он или нет? Обязателен экран успеха с анимацией и номером брони.

**7. Регистрационные формы**  
«Введите имя, фамилию, email, телефон» — в Telegram это нонсенс. Имя и фото уже есть. Телефон берётся через `requestContact()` одним тапом. Форма с полями убивает конверсию.

#### Что обязательно должно быть

- **Цена и длительность на каждой карточке** — не «уточняйте», не «от». Конкретно: 1 500₽ / 1ч 30м.
- **Прогресс-индикатор** в флоу записи — «Шаг 2 из 3» или progress bar вверху экрана.
- **Политика отмены** видна до подтверждения — не после. Прозрачность снимает тревогу.
- **Skeleton-экраны** при загрузке — не spinner. Skeleton воспринимается как «грузится», spinner как «завис».
- **Фотографии реальных работ** мастера, не стоковые. Стоковое фото ногтей при мастере по бровям — доверие рушится мгновенно.
- **Haptic feedback** на ключевых действиях: выбор слота, подтверждение. Делает интерфейс живым.

#### Что бесит пользователей

- Бесконечный скролл без якорей — потерялся в каталоге, нет способа вернуться к нужной категории
- Горизонтальный свайп у края экрана на iOS закрывает Mini App — теряешь всё что выбрал
- Всплывающие окна поверх флоу бронирования — акция, подписка, оценка — в самый неподходящий момент
- «Уточняйте цену у мастера» вместо конкретной суммы — раздражает и отталкивает
- Разные стили элементов на разных экранах — выглядит как несколько несвязанных приложений

### 10.2 Маркетолог

#### Что заставляет записаться

**1. Социальное доказательство «на входе»**  
Рейтинг мастера (звёзды + число отзывов) должен быть виден ещё на карточке в каталоге, до перехода в профиль. «⭐ 4.9 (127 отзывов)» работает как сигнал доверия. Клиент принимает решение ещё до открытия карточки.

**2. Портфолио как первый экран, не форма**  
Исследование показало: все конкуренты прячут галерею. Мы ставим работы мастера на главный экран. Клиент «влюбляется» в работу → хочет такой же результат → запись естественна. Это перевёртывает воронку: не «выбери услугу», а «хочу вот это».

**3. Срочность без манипуляции**  
«Осталось 2 слота на этой неделе» — честная информация, которая стимулирует действие. Работает если это правда (синхронизировано с реальным расписанием). Фейковые таймеры убивают доверие навсегда.

**4. «Записаться снова» с предзаполнением**  
После первого визита кнопка «Записаться снова» в истории записей автоматически подставляет ту же услугу, того же мастера, предлагает ближайший свободный слот. Один тап — и клиент снова в записи. Это снижает время принятия решения с минут до секунд.

**5. Фото «до и после»**  
На экране детали услуги слайдер «до/после» — самый конвертирующий элемент в бьюти. Клиент видит реальный результат, а не описание. Особенно работает для коррекции бровей, восстановления ногтей, окрашивания.

**6. Первый визит без риска**  
«Отмена бесплатно за 24 часа» на видном месте снимает барьер нерешительности. Клиент боится «попасть» — показываем что выход есть.

#### Элементы, повышающие конверсию (по убыванию важности)

| Элемент | Почему работает |
|---|---|
| Реальные фото работ на карточках | Бьюти = визуальный выбор, текст не убеждает |
| Цена и время без скрытых условий | Прозрачность снимает тревогу перед нажатием |
| Рейтинг + число отзывов в каталоге | Социальное доказательство до клика |
| «Осталось N слотов» (если правда) | Дефицит стимулирует немедленное действие |
| Экран успеха + «Поделиться» | Вирусность: клиент делится в Telegram, приходят новые |
| Кнопка «Записаться снова» | Снижает порог повторной записи до одного тапа |
| Автонапоминание за 24ч + 2ч | Снижает no-show, мастер не теряет деньги |
| Слайдер «до/после» на услуге | Самый убедительный визуальный аргумент в бьюти |
| Депозит при записи | Клиент «вложился» — вероятность прихода выше |

#### Чего не делать маркетологу

- Поп-апы с акциями в момент записи — конкурируют с основным действием, конверсия падает
- «Подпишитесь на рассылку» до того как клиент хоть раз записался — слишком рано
- Скрытые условия (цена «от», «уточняйте») — клиент закрывает и не возвращается
- Искусственные таймеры «скидка сгорит через 10 минут» — разоблачаются и разрушают доверие

### 10.3 Telegram-разработчик

#### Ограничения, о которых нужно знать заранее

**1. Payments: физические услуги ≠ Telegram Stars**  
Telegram Stars (`XTR`) — только для цифровых товаров. Для записи к мастеру (физическая услуга) нужен `openInvoice()` с реальным платёжным провайдером: ЮKassa, CloudPayments, Stripe. Без регистрации провайдера оплата недоступна. Это блокер для соло-мастеров без ИП — нужно предусмотреть режим «без оплаты» как fallback.

**2. `requestContact()` работает один раз**  
После того как пользователь дал номер телефона, повторный запрос не вызывает диалог — просто ничего не происходит. Сохранять телефон в базу при первом же согласии, не рассчитывать на повторное получение.

**3. Горизонтальный свайп на iOS = закрытие приложения**  
На iPhone свайп от левого края — системный жест «назад». Любой горизонтально скролящийся элемент (галерея, слайдер) у самого края вызывает закрытие Mini App. Правило: отступ от левого/правого края не менее 16px, горизонтальные галереи — внутри контейнера с padding.

**4. `disableVerticalSwipes()` — защита флоу записи**  
По умолчанию свайп вниз закрывает Mini App. На шагах бронирования это катастрофа. Вызывать `disableVerticalSwipes()` при входе в флоу записи и `enableVerticalSwipes()` при выходе.

**5. `ClosingConfirmation` — не забыть включить**  
На шагах 2–4 флоу записи включать `enableClosingConfirmation()`. Пользователь случайно нажал крест — получает диалог «Вы уверены?». Без этого — молчаливая потеря прогресса.

**6. `CloudStorage` — синхронный API, лимит 1024 ключа**  
`CloudStorage` хранит строки по ключу, не объекты. Сериализовать в JSON перед записью. Лимит: 1024 ключа × 4096 символов. Для черновика записи хватит с запасом. Не использовать для хранения фотографий.

**7. Размер Mini App: тема Telegram не равна вашей теме**  
`themeParams` даёт цвета текущей темы Telegram (bg, text, button, hint, link). Но мастер может захотеть брендовые цвета. Решение: брендовые цвета применять только к акцентам (кнопки, значки цен), фон и текст — из `themeParams`. Тогда работает и в тёмной, и в светлой теме.

**8. Safe Area Insets**  
На iPhone с Dynamic Island и на Android с жестовой навигацией нижняя часть экрана перекрыта системными элементами. `MainButton` Telegram сам учитывает safe area, но кастомные элементы внизу (bottom sheet, sticky кнопка) — нет. Использовать CSS `env(safe-area-inset-bottom)` для padding.

**9. Загрузка изображений — главный враг производительности**  
Портфолио мастера — десятки фото. Без оптимизации: белые квадраты вместо изображений и 5 секунд загрузки. Обязательно: WebP формат, lazy loading, thumbnail 300px для сетки и полное фото только при открытии. CDN обязателен.

**10. Нет доступа к файловой системе**  
Загрузка фото мастером возможна только через `<input type="file">` или через бот (мастер отправляет фото боту, тот сохраняет). Нет `FileSystem API`, нет drag-and-drop. Учитывать при проектировании панели мастера.

#### Что работает хорошо

| Возможность | Применение в нашем проекте |
|---|---|
| `MainButton` | CTA на каждом шаге — «Выбрать время», «Подтвердить», «Оплатить» |
| `HapticFeedback` | Выбор слота, подтверждение брони, звёзды отзыва |
| `BackButton` | Нативная навигация назад — не нужно делать свою |
| `initData` + HMAC-SHA256 | Верификация пользователя на сервере без логина |
| `themeParams` | Автоматическая адаптация к тёмной теме |
| `showPopup()` / `showConfirm()` | Нативные диалоги отмены брони — не делать кастомные модалки |
| `openTelegramLink()` | Переход в чат мастера одним тапом из приложения |
| `shareMessage()` (Bot API 8.0+) | Клиент делится своей записью — виральность |
| Home Screen shortcut (Bot API 8.0+) | «Добавить на главный экран» — иконка мастера на телефоне клиента |
| Full-screen mode (Bot API 8.0+) | Иммерсивный просмотр портфолио без Telegram-хрома |

#### Что лучше не делать

- **Кастомный back button** вместо `BackButton` — пользователь привыкает к нативному, своя кнопка вызывает путаницу
- **iframe внутри Mini App** — Telegram блокирует большинство iframe по CSP
- **Автоплей видео** — часть пользователей открывает в тихом режиме, видео без звука без предупреждения раздражает
- **Открытие внешних ссылок без предупреждения** — Telegram показывает диалог «открыть браузер?», это прерывает флоу
- **WebSocket без переподключения** — Mini App может быть «заморожен» при сворачивании Telegram. При возврате WebSocket мёртв. Нужна логика reconnect + отображение «Обновляем данные...»
- **Анимации без `prefers-reduced-motion`** — часть пользователей отключает анимации на системном уровне; тяжёлые Lottie без проверки этой настройки вызывают раздражение и могут вызывать дискомфорт у людей с вестибулярными нарушениями

### 10.4 Общий список рекомендаций

#### Критически важно (без этого не запускать)

1. **Фото на каждой карточке услуги** — реальные работы мастера, не стоковые
2. **Цена и длительность видны без клика** — на карточке, не внутри
3. **Сохранение черновика** в `CloudStorage` — восстанавливать при следующем открытии
4. **`ClosingConfirmation` + `disableVerticalSwipes()`** на шагах записи
5. **Экран успеха с анимацией** после подтверждения — пользователь должен понять что записался
6. **Safe area insets** для нижних элементов — особенно на iPhone с Dynamic Island
7. **Skeleton-экраны** при загрузке, не spinner
8. **Верификация `initData` на сервере** (HMAC-SHA256) — защита от фейковых запросов

#### Важно для конверсии

9. **Рейтинг и число отзывов** видны в каталоге — не только внутри профиля
10. **«Осталось N слотов»** на экране выбора времени — честный дефицит
11. **Кнопка «Записаться снова»** в истории с предзаполнением
12. **Слайдер «до/после»** на экране детали услуги (хотя бы для ключевых услуг)
13. **`HapticFeedback`** на выборе слота и подтверждении
14. **Политика отмены** видна до нажатия «Подтвердить»

#### Для второй версии

15. **Full-screen режим** для портфолио (Bot API 8.0+)
16. **Home Screen shortcut** — предложить добавить мастера на главный экран после первой записи
17. **`shareMessage()`** на экране успеха — клиент делится записью
18. **Программа лояльности** — баллы за каждый визит
19. **Слайдер «до/после»** для всех услуг
20. **Push-уведомления** через бот: напоминание за 24ч и 2ч (не email)

#### Чего избегать

- Горизонтальный скролл у краёв экрана на iOS (закрывает Mini App)
- Регистрационные формы с полями — имя из Telegram, телефон через `requestContact()`
- Поп-апы с акциями в момент флоу записи
- Spinner вместо skeleton
- Скрытые условия по цене («уточняйте»)
- WebSocket без логики reconnect
- Кастомный back button вместо нативного `BackButton`
